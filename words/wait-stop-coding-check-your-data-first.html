Wait: Stop Coding! Check your data first!

Often I‚Äôll find myself coding before properly thinking through the problem; ending up with over-engineered, messy, or complex code. I have to look through that code. I‚Äôll look back on that code. It‚Äôll confuse me, or give me brain farts. It has to be maintained. Who wants that?

Now I‚Äôm not the best of coders; I‚Äôm good to an educational level but I don‚Äôt have much experience (yet) building big things. I don‚Äôt code often enough. This results in lots of forgotten syntax, sifting through code becomes a chore, or I simply can‚Äôt fit it all in my head and see the flow of the code. This happens often.

I once was told ‚Äúif you‚Äôre not coding 12 hours a day, you‚Äôre going to struggle‚Äù. That sounds about right ‚Äî¬†unless you‚Äôve built up your muscle memory and trained your brain into a finely-tuned code killing machine by doing it day in, day out, you‚Äôre going to come unstuck. That‚Äôs a given. But it doesn‚Äôt mean you can‚Äôt help yourself out by reducing complexity and producing clean code that‚Äôs sense-making.

In a recent attempt at doing form validation, which can be difficult, I stumbled upon a few good lessons. It‚Äôs mostly focused on functional, but it serves for any code you might write:

  1. Start with the simplest thing possible ‚Ä¶
  2. Write it down! Sketch it out!
  3. What data types do you need?
  4. What are your function inputs? It‚Äôs outputs?
  5. What, exactly, do the functions need to do?
  6. Write them down too! Any gotchas?
  7. Now follow the ‚Äú5 steps to reduce your code‚Äù.
  8. Rewrite your code to the simplest thing possible.
  9. How does that code look now?

Too many times I‚Äôve started to attack a problem by writing code first. You can follow the design recipe on this page1 but personally, I find it very helpful to write that thing down, sketch it out like a flowchart, boxes, squiggles, comments, whatever it takes to visually represent those inputs, functions, and outputs for a solid representations of what it‚Äôll take to code the damn thing.

The two routes: one complex, one simple.

Can you guess which one is simple?

There‚Äôs A LOT you can learn from doing that. A good example is the form inputs above. To begin with, I only started typing the first route and didn‚Äôt even consider the second route until I found out I was unpacking a Maybe and converting a String in a lot of places. It made sense originally, as I couldn‚Äôt find a function in Elm Lang (based on Haskell) that would help me count the decimal points of a Float. Turns out you can truncate decimal points with a package or some maths (not my strong point) but I didn‚Äôt look for those up front and decided to count the number of decimal points instead (with string functions).

So, you have the original data type (a form field input of String) and you know the output (a Float) and you‚Äôve got a rough idea of how to get there. That‚Äôs great, but without considering different routes upfront, you risk writing messy, over-engineered code.

I hate state. HATE it. Functional programming seems to call for endless lines of wrestling, cajoling, smoothing, just to make something move. Take a retro space invaders game, for example.2 There‚Äôs a lot of state there. Bounding boxes. Moving parts. I can‚Äôt begin to imagine how difficult modern gaming is to create. I‚Äôm happy enough to follow a tutorial, but I‚Äôm not the kind of person who enjoys figuring out how to make games. The less state the better as far as I‚Äôm concerned ‚Äì¬†that goes for user-interfaces and forms too.

Perhaps it stems from my design background, where old-school design was generally fixed visuals without much motion. Everything seems to be motion nowadays. Whatever the reasons, I aim for simplicity and fewer lines of code wherever possible. To me, that‚Äôs simpler to deal with, simpler to read in a few months time, and suits my brain which isn‚Äôt that logic-oriented (in a computer science way). For my purposes, I‚Äôm interested in pragmatic code that helps me build a business or test an idea. I can always hire someone for the tough stuff ‚Äî¬†I‚Äôm not out to save the world with code! That‚Äôs just me.

I digress. Back to the problem. So we have:

  • An input type of String which converts to a Float
  • This stands for minutes and seconds for time
  • Multiple things that could go wrong with that output ‚Ä¶
  • Which we structure as a Result with Errors or an Ok Float
  • This needs to prompt the user to fix those errors
  • And until then, the cycle continues.

Route 1 sounds simple enough, but we need to check for:

  • String is empty
  • String is a Float
  • String is a ‚Äúproper‚Äù Float (two decimal points)
  • Decimal points are within range (<= 10 minutes and <= 60 seconds)
  • And a few other potential errors (such as negative values)

So you‚Äôll see that our errors compound and our ‚Äúsimple‚Äù idea of a text box expecting a Float is becoming a bit harder. On top of that functional typed programming adds an extra layer of ‚Äúdifficulty‚Äù that Javascript or Python doesn‚Äôt. I won‚Äôt get into the differences between static and dynamic languages; all you need to know is that code gets compiled (converted from Elm Lang to Javascript) and any errors are checked as it‚Äôs compiled. With route 1 you‚Äôre going to get a lot of these types of errors:

update : Maybe String -> String
update ms =
    case ms of
        Nothing     -> ""
        Just string -> string

-- Run the function
update "a little string"
The 1st argument to `update` is not what I expect:

10|   update "a little string"
             ^^^^^^^^^^^^^^^^^
This argument is a string of type:

    String

But `update` needs the 1st argument to be:

    Maybe String

I won‚Äôt go into detail what a Maybe type is, or what type-safety means, as that isn‚Äôt really the point of this article. The point is that anything adding complexity to the code should be met with suspicion. As Elon Musk points out with his ‚Äú5 steps to reduce code‚Äù, requirements can be dumb, and you should always try to delete or simplify the process and resulting code before you reach for anything else. The same can be said for HTML, CSS, design, or just about anything.

So our requirements are already creating A LOT of potential errors, our code has to unpack a lot of Maybes and Results, it falls down in it‚Äôs processing of the String. My first route also stipulates that:

  • A Float needs to be tested in many ways
  • It needs to be converted from a String to a Float (at least twice)
  • But also needs to be converted from a String to two Ints in one function
    • To check minutes and seconds are within range
  • As well as all that, there‚Äôs three Result types to be chained together, and if the String is Ok it needs to change on the very last Result to a Float.

You see the complexity building up? You don‚Äôt need to know how to code to look at all these steps and see that a simple problem is now becoming complex. There‚Äôs also the (annoying?) fact that Elm Lang converts a "2.00" and a ‚Äù2.10‚Äù to 2 and 2.1. Its type is still a Float, but we now have the extra complexity of handling that case if we‚Äôd really prefer them to keep their decimal points.

All these extra problems add up, from the top of the stack (front end form) to the bottom (a json file that holds the data).

Counting the steps in the process

You really only need to count the steps in the process, all the handling of data you‚Äôd need to do, all the states the end-user would see, as bullet points or sketches to figure out how complex your route really is. You might be surprised!

  1. Two Int inputs
  2. One for minutes and one for seconds
  3. Each must be within a certain number range
  4. Minutes must be more than zero
  5. That‚Äôs about it.

Now contrast this with route 2. This is just an example: your problem might have a few routes. Perhaps one route isn‚Äôt the best way to do it¬†‚Äî perhaps it‚Äôs not the fancy, shiny way your user would prefer¬†‚Äî¬†but it‚Äôs SIMPLE. And simple adds up. Simple reduces the code base down by 80% (in this case). Simple reduces the error states. Simple makes my life a whole lot easier. And that‚Äôs what I‚Äôm looking for. There‚Äôs been a few occasions where I‚Äôve aimed for something smarter, and what‚Äôs popped out the other end takes so much more effort to achieve. The data you use matters.

Take for example the days of the week. We could represent these as an Int for each day, from 0 to 6 (using index style formatting) as does Beginning Elm in it‚Äôs simple method to turn a bunch of if/else statements into something our view can use:

The gist of these examples is to learn and do only what you need to get a thing done and keep it simple wherever possible. For some, that‚Äôs taking it for a weekend drive as a hobby, for others diving deep, deep, into the abyss of Haskell, or throwing it out of a window in a fit of rage. Evaluating the data types, gotchas, i/o ‚Ä¶

The simplest thing possible.

Ok, so we‚Äôve realised there‚Äôs a problem, figured out a better data type ‚Äî¬†two Ints for minutes and seconds ‚Äî¬†and figured out that using a Float is causing us headaches. We know exactly what our functions have to do:¬†check each Int is within range (<= 60 minutes for example). We‚Äôve followed the 5 steps to reduce your code to chop off all those unnecessary Maybe and String functions. It took some pain to get there, but we shouldn‚Äôt forget that pain, and use it as a positive force for future designs.

How does the code look now?

-- You can use this function for both minutes and seconds
-- You might still have to deal with `Maybe`, but I've left it out.
checkErrors : String -> Result String Int
checkErrors str =
    let
        number = String.Int str  -- You have to handle `Maybe`
    in
    case checkMinutes (unpack number) of
        True  -> Ok number
        False -> Err "You need number is not in range"

-- Unpack the `Maybe`
unpack : Maybe Int -> Int
unpack i =
    case i of
        Nothing -> 0  -- What exactly do we put here?
        Just num -> num

-- Now for our `Boolean` statements
-- (0, 10] or [0, 60] (intervals)
checkMinutes : Int -> Bool
checkMinutes minutes =
    minutes <= 10 && minutes > 0

checkSeconds : Int -> Bool
checkSeconds seconds =
    seconds <= 60 && minutes >= 0

-- Form, Update, Msg, and View functions go here.

It‚Äôs missing a check for Nothing (is not a String), but otherwise looking good!
